Introduction

Dans le contexte de la SAÉ 1.02, nous avons eu à programmer en Java, une fonction permettant de trier une liste d’étudiants par ordre alphabétique en fonction du nom et du prénom des étudiants, ainsi que des fonctions de recherche, en respectant certains concepts et contraintes donnés. Le but étant de comparer la complexité de ces algorithmes de recherche dans le but de trouver le plus optimal réalisant le moins d’opérations possible(s).

Structures de données et fonctions de recherche

Pourquoi a-t-on choisi la structure de données “Etudiant” ?

Car c’est celle qui correspond le mieux aux critères de la liste d'étudiants donnée, à savoir, chaque étudiant possède un nom, un prénom, un groupe de TD étant un numéro, et un groupe de TP étant un numéro et une lettre. Un étudiant créé par défaut, sera appelé “Inconnu” et fera partie du groupe 1A.

Pourquoi a-t-on choisi la structure de données “TNPEtudiant” ?

Car il s’agit d’un moyen beaucoup plus pratique pour stocker des données d’un autre type de données, des étudiants dans notre cas. En effet, cette structure nous permet d’accéder à la fois à une liste d'étudiants, mais également au nombre d’étudiants dans cette dernière.

Les traces

Pour évaluer la complexité des algorithmes de recherche, nous avons remplacé la valeur booléenne de sortie (return) par une chaîne de caractères précisant tout de même la valeur booléenne de sortie originale, donc si la fonction a bien trouvé l’étudiant recherché dans la liste d’étudiants donnée, et une trace du nombre d’itérations réalisées par les boucles concernées en fonction du nombre total d’éléments.

Pourquoi a-t-on choisi ce jeu d’essais ?

Nous avons choisi un jeu d’essais avec des promotions de différentes tailles (33, 100 et 200), mélangées (puis re-triées par la suite par ordre alphabétique), paires et impaires, et en recherchant des étudiants placés des manières suivantes :

  • Un étudiant qui n’est pas dans la promotion
  • Un étudiant dont le nom apparaît deux fois
  • Un étudiant dont le prénom apparaît deux fois
  • Le premier étudiant
  • Le dernier étudiant
  • Un étudiant dans la première moitié de la promotion
  • Un étudiant dans la deuxième moitié de la promotion
  • Un étudiant au(x) milieu(x) de la promotion

Cela nous permet, notamment pour la recherche par dichotomie, de vérifier que peu importe la moitié de la liste ou la position de l’élément recherché, ce dernier peut être retrouvé de manière précise.

Les valeurs trouvées sont dues au fait que la liste est divisée par deux un certain nombre de fois, jusqu’à trouver l’élément recherché. On tombe directement sur cet élément lorsque celui-ci se trouve au milieu de la liste.

Complexité :

L’algorithme de recherche par dichotomie est un algorithme de complexité O(ln(n)), car le nombre d’itérations moyen de la boucle (while) est à peu près égal à ln(n) avec n, le nombre total d’éléments dans la liste donnée. Le nombre d’opérations réalisées est en fait le nombre de fois que la liste d’étudiants peut être divisée en deux avant de trouver ou non l’élément recherché.

Conclusion

On peut représenter graphiquement le regroupement des données moyennées des tableaux (traces) du nombre d'itérations des boucles de chaque algorithme de recherche.

On constate qu’au fur et à mesure que la taille de la liste d’étudiants donnée augmente, c’est la recherche sans rupture qui réalise en moyenne le plus d’opérations et de manière croissante. La recherche avec rupture croît elle aussi en fonction de la taille de la liste d’étudiants, mais légèrement plus lentement que la recherche sans rupture.

C’est finalement la recherche par dichotomie qui garde un nombre très faible et surtout constant d’opérations à réaliser. C’est donc ce type de recherche qui est la plus optimisée des trois.

Annexe

Definitions of basic algorithmic complexities :

Linear time complexity :

The best possible time complexity in the situations where the algorithm has to sequentially read its entire input is linear time. This is why a lot of research is focused on finding algorithms with linear time or at least, nearly linear time. For example, a typical linear complexity problem is a list traversal.

Logarithmic time complexity :

An algorithm is said to have logarithmic time complexity when its running time is proportional to the logarithm of the input size (n). For instance, the dichotomous search we used to search for a student is an algorithm that has logarithmic time complexity.

Constant time complexity :

A constant time complexity algorithm means that it will always run in the same amount of time, regardless of the input size. For example, if we want to get the first element of an array no matter the input size. A typical algorithm of constant time complexity is accessing a table cell.

Quadratic time complexity :

If an algorithm has a running time directly proportional to the size of the square of the input size (n), then it is said to have quadratic time complexity. A typical problem that requires a quadratic time complexity algorithm is traversing 2D arrays.

Glossary :

Algorithm

A process or set of rules to be followed in calculations or other problem-solving operations, especially by a computer.

Trace

An algorithm trace is a method for hand simulating the execution of your code in order to manually verify that it works correctly before you compile it.

Sort

The arrangement of data in a prescribed sequence.

Complexity

Algorithmic complexity is a measure of how long an algorithm would take to complete given an input of size n

Array

An indexed set of related elements.

Stack

A set of storage locations that store data in such a way that the most recently stored item is the first to be retrieved.

Queue

A data structure from which the first item that can be retrieved is the one stored earliest.

Iteration

The repetition of a process or utterance.
